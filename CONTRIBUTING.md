# Contributing to the ISO/IEC 27701:2025 PIMS Toolkit

Thank you for your interest in contributing to this toolkit! This project aims to be the most comprehensive, practical, and freely available ISO 27701 implementation resource for privacy and GRC professionals worldwide.

---

## Ways to Contribute

### 1. Report Issues
If you find an error, outdated information, or missing content:
- Open a GitHub Issue
- Use the title format: [BUG] or [IMPROVEMENT] followed by a brief description
- Describe the issue clearly, referencing the specific file and section

### 2. Suggest Improvements
Have an idea for a new document, a better template, or an additional worked example?
- Open a GitHub Issue with the title format: [SUGGESTION] Your idea
- Describe the improvement and why it would add value to the toolkit

### 3. Submit a Pull Request
To contribute a document or fix directly:
1. Fork the repository
2. Create a branch named: feature/your-description or fix/your-description
3. Make your changes following the style guide below
4. Submit a Pull Request with a clear description of your changes
5. Link any related Issues in the PR description

---

## Style and Quality Guidelines

All documents must meet the following standards:

### Document Structure
- Include a document header with: Document ID, Version, Date, Owner, Classification, Review Date
- Use consistent heading levels (## for sections, ### for subsections)
- Include a Record of Changes table at the end
- Include the footer citation: *ISO/IEC 27701:2025 PIMS Toolkit | [Document Name] | [Document ID]*

### Document IDs
Follow the existing convention:
- Policies: PIMS-POL-NNN
- Context: PIMS-CTX-NNN
- Leadership: PIMS-LDR-NNN
- Planning: PIMS-PLN-NNN
- Support: PIMS-SUP-NNN
- Operations: PIMS-OPS-NNN
- Performance: PIMS-PER-NNN
- Improvement: PIMS-IMP-NNN
- Supplementary: PIMS-SUP-0NN (10 series)

### Content Quality
- Content must be substantive, not placeholder filler
- Tables should be used for structured data (registers, RACI matrices, checklists)
- ISO 27701 clause references should be included where applicable
- GDPR article references should be included where relevant
- Language should be professional and accessible (plain English preferred)
- Avoid jurisdiction-specific assumptions unless explicitly scoped

### File Naming
- Use UPPERCASE-KEBAB-CASE.md for all Markdown files
- Use lowercase_snake_case.py for Python scripts
- Be descriptive: DATA-SUBJECT-RIGHTS-PROCEDURE.md not DSRP.md

---

## Accuracy Standards

This toolkit is used by real organisations for real compliance purposes. All contributors must:

- Verify factual claims against the ISO/IEC 27701:2025 standard (available from ISO.org)
- Verify regulatory references against current legislation (GDPR, PECR, etc.)
- Do not include legal advice — include appropriate disclaimers on legal/contractual templates
- Do not claim to represent official ISO guidance
- Flag any content that may be jurisdiction-specific

---

## What We're Looking For

High-priority contributions include:

- **New jurisdiction guides** (e.g., Brazil LGPD, India DPDPA, Singapore PDPA, Canada PIPEDA)
- **Sector-specific variants** (healthcare, finance, HR, education)
- **Additional worked examples** using the fictional NFS organisation
- **Translations** of key documents into other languages
- **Improvements to existing documents** — especially gap analysis tools and DPIA templates
- **Bug fixes** — incorrect cross-references, outdated regulatory references

---

## Code of Conduct

This project is open to contributors of all backgrounds and experience levels. We expect all contributors to:
- Be respectful and constructive in all communications
- Focus on making the toolkit better, not on personal opinions
- Provide evidence-based suggestions when disagreeing with existing content
- Follow GitHub's community guidelines

## Questions?

Open an Issue on GitHub or contact the maintainer through the repository's contact information.

Thank you for helping make privacy compliance more accessible!
